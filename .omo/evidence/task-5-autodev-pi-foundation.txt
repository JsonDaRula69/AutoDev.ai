## 4. Context Injection Verification

`extensions/autodev/context.ts` implements context injection via the
`before_agent_start` event (not DefaultResourceLoader, since the extension
cannot construct the session loader). Strategy:

1. `loadContextFiles(projectRoot)` reads AGENTS.md, CONTEXT.md, and every
   `.md` under `.autodev/memory/` (projectbrief, techContext, activeContext,
   harbor-log). Missing files are silently skipped.
2. `renderContextBlock(files)` wraps each file in an HTML comment marker
   and joins them under a `# AutoDev Project Context` heading.
3. `augmentSystemPrompt(event, cwd)` appends the block to the chained
   system prompt. Returns the original prompt unchanged when no context
   files are present (fresh checkout safe).

This augments pi's default context loading rather than replacing it — the
MUST NOT constraint "Do NOT break pi's default context loading" is satisfied.

Verified: AGENTS.md (10203 bytes), CONTEXT.md (7623 bytes), and 4 memory
files exist at the project root and are discovered by loadContextFiles.

## 6. comment-checker — Slop Patterns

`stripSlop(source)` is a pure function detecting 5 AI-slop comment patterns:

| Pattern | Regex | Action |
|---------|-------|--------|
| restates-name | `// This (function\|method\|class\|module) \w+` | strip comment-only line |
| narrative-filler | `// (First\|Next\|Then\|Finally),? we ` | strip comment-only line |
| obvious-observation | `// Note that ` | strip comment-only line or trailing comment |
| vague-todo | `// TODO: implement` | strip (suggest concrete TODO) |
| declaration-restates-name | `/** This is a \w+` | strip doc comment |

The `tool_result` handler inspects write/edit results, runs stripSlop on
the written content, and emits `ctx.ui.notify` warnings. The write is
never blocked — slop is stripped and good code passes through.

## 7. intent-gate — Two Analyzers

**analyzeIssueIntent(text)** — for Nemo triage, before Cynefin:
- Classifies: bug, feature, refactor, question, chore, ambiguous
- Returns: intent, confidence (0..1), signals[], suggestedCynefin, probingQuestions[]
- Keyword+structure based, deterministic, no LLM call

**analyzeOnboardingIntent(text)** — for Harbor Master:
- Surfaces hiddenIntentions: scale, migration, cost-pressure, time-pressure, hidden-complexity
- Detects stake: low/medium/high/critical/unknown
- Detects technicalDepth: non-technical/mixed/technical
- Returns probingQuestions[] calibrated to stake level

## 8. notepad — 5 Storage Mappings

| Function | Backend | Written? |
|----------|---------|----------|
| storeLearning | ctx_memory ARCHITECTURE | no (caller invokes ctx_memory) |
| storeDecision | loreguard suggest_lore (ADR draft) | no (caller invokes suggest_lore) |
| storeIssue | ctx_memory CONSTRAINTS | no (caller invokes ctx_memory) |
| storeVerification | .omo/evidence/<name>.txt | yes (writes file immediately) |
| storeProblem | .autodev/research/<name>.md | yes (writes file immediately) |

Each returns a `StorageDescriptor` with kind, backend, target, content,
written flag, and note. File-writing backends create parent dirs
recursively and append the correct extension (.txt / .md).
