## Config decisions

### `.pi/settings.json`
- `defaultProvider: "ollama-cloud"`, `defaultModel: "glm-5.2:cloud"` — AutoDev's triage/plan/deploy model per ROADMAP model routing.
- `compaction.enabled: false` — Magic Context owns context management; pi's built-in compaction would conflict.
- `enabledModels` lists both valid models for Ctrl+P cycling.
- `extensions: ["./extensions/autodev"]` — registers the AutoDev extension (created in T5).
- `$schema` references pi's settings schema for editor validation.

### `.pi/magic-context.jsonc`
- `embedding`: VoyageAI (`voyage-3-large` model) via `openai-compatible` provider; API key referenced as `${VOYAGE_API_KEY}` env var (installer T19 prompts at deploy time).
- `dreamer.enabled: true`, model `ollama-cloud/glm-5.2:cloud`, fallback `ollama-cloud/deepseek-v4-pro`.
- `dreamer.user_memories.enabled: true` — required per task spec.
- `dreamer.pin_key_files.enabled: true` — required per task spec.
- `memory.git_commit_indexing.enabled: true` — required per task spec.
- `sidekick.enabled: true` — required per task spec.
- `historian` + `dreamer` + `sidekick` all use `ollama-cloud/glm-5.2:cloud` primary with `deepseek-v4-pro` fallback.
- `$schema` references Magic Context's schema for editor validation.

### `.pi/auth.json` (template, gitignored)
- Documents pi's auth resolution order per providers.md.
- `ollama-cloud` provider key template references `$OLLAMA_CLOUD_API_KEY` env var.
- All comments use `_comment_N` keys (JSON doesn't support native comments; this is a `.json` file not `.jsonc`).
- Explicitly notes the installer (T19) populates this file — no credentials extracted during development.

### `tsconfig.json`
- `target: ES2022`, `module: ESNext`, `moduleResolution: bundler` — pi uses jiti for TS loading without compilation; bundler resolution matches the runtime.
- Strict mode suite: `strict`, `noUncheckedIndexedAccess`, `exactOptionalPropertyTypes`, `verbatimModuleSyntax`, `isolatedModules`.
- `types: ["bun-types"]` — Bun runtime types.
- `include: ["extensions/**/*.ts", "test/**/*.ts", "src/**/*.ts"]`.

### `.autodev/config/models.json`
- Simple JSON array of the 2 valid model strings, matching the task spec exactly.

## Post-write architectural review

1. **Single responsibility?** Each config file owns one concern: package.json = manifest, settings.json = pi runtime config, magic-context.jsonc = memory config, auth.json = credentials, tsconfig.json = TS compiler config, models.json = model list. No file does two things.
2. **Boundary purity?** Config files are boundary artifacts — they declare, they don't contain logic.
3. **Escape hatches?** None in config. Test file uses `(error as Error).message` in the catch block — necessary for typed error handling in the import-check test.
4. **File size?** All files well under 250 pure LOC ceiling (max is 106 in the test file).
5. **Tests?** Import resolution locked by `test/import-check.ts` — would fail if SDK exports changed.
