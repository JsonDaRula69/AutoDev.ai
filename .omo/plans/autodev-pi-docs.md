The `pi-foundation` branch was created from `main` as a fresh start — all commits (docs + implementation) land here. `main` is frozen and will not receive any pushes during this work. Upon completion of both plans (docs + implementation), `main` will be deprecated and `pi-foundation` will become the new `main` branch (via branch rename or fast-forward merge at the user's discretion).

## TL;DR (For humans)

**What you'll get:** Four comprehensive documentation files — a new README.md, a new ARCHITECTURE.md, a new STRUCTURE.md, and a new ROADMAP.md — that describe how the pi-based AutoDev works from top to bottom. These docs are the design specification that the pi-foundation implementation plan follows. ARCHITECTURE.md covers what IS built (27 sections); ROADMAP.md covers what is NOT yet built (future waves). They cover: what AutoDev is, the Nautilus crew, the pi runtime foundation, the autonomous workflow from issue to merge, the extension architecture, Magic Context integration, Loreguard, Discord, guardrails, the debate protocol, setup instructions, and the project directory structure.

**Why this approach:** The docs come before the implementation so they serve as the source of truth. The pi-foundation plan references these docs as the specification. Writing the docs first forces us to think through the complete design — every component, every interaction, every edge case — before writing a single line of code. This is the Aronnax Method: design before building.

**What it will NOT do:** No implementation code. No pi extension files. No agent definitions. No config files. Only documentation (.md files). The existing README.md, .autodev/ARCHITECTURE.md, and STRUCTURE.md are replaced (the old OpenCode-based versions are superseded).

**Effort:** Medium — 7 todos, mostly parallel, producing 4 large markdown files.
**Risk:** Low — documentation only, no code changes. The main risk is inaccuracy: if the docs describe something that can't actually be built on pi, the implementation plan will fail. Mitigated by the extensive pi research already completed (SDK capabilities, extension API, subagent patterns, Magic Context Pi extension).
**Decisions I made for you:** (1) The docs are written for the NEW pi-based design, not the old OpenCode-based one. (2) README.md is user-facing (setup, usage, crew overview). ARCHITECTURE.md is developer-facing (system design, component interactions, data flow). STRUCTURE.md is the directory map and reference catalog. (3) The docs reference the pi-foundation plan for implementation details but don't duplicate the plan's todo-level granularity. (4) The existing .autodev/ARCHITECTURE.md and STRUCTURE.md are replaced by root-level ARCHITECTURE.md and STRUCTURE.md (the .autodev/ versions were OpenCode-specific).

Your next move: approve, then execute. Once these docs are written, the pi-foundation plan can begin.
