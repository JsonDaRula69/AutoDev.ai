# docs-2tier-system - Work Plan

## TL;DR (For humans)

**What you'll get:** A two-tier documentation search system. Framework dependency docs (pi, Magic Context, Bun, Ollama, Discord, LSP, etc.) live in a central store shared across all projects on your machine. Project-specific docs live in each project. Search returns results from both, ranked by relevance. Docs are fetched fresh from upstream sources via git sparse-checkout or llms.txt endpoints, never shipped stale in the repo.

**Why this approach:** Central + project tiers avoid re-downloading and re-embedding the same dependency docs for every project. Seeding from upstream sources (not shipping a static collection) means you always have the latest docs — AutoDev isn't responsible for maintaining a doc collection that goes stale. Per-source exclude patterns filter out legal/ToS/privacy/changelog junk so only development-relevant docs are indexed.

**What it will NOT do:** No docs shipped in the npm package or repo. No format converters for non-Markdown sources (pip/RST, git/AsciiDoc, SQLite/HTML, Telegram/HTML, tmux/mdoc) — those sources are deferred until converters are built. No changes to embedding providers, chunking, or similarity math. No hardcoded source URLs in code — sources are config-driven.

**Effort:** Large
**Risk:** Medium — touches the docs module core, installer, doctor, CLI, and init; but the building blocks (embeddings, chunking, vector store) stay unchanged.
**Decisions to sanity-check:** Central DB co-located with corpus via `getAgentDir()/../docs-corpus/vectors.db`. Seeding source list defaults to empty. 6 non-Markdown sources deferred (pip, git, SQLite FTS5, SQLite general, Telegram, tmux). `git rm --cached` (not disk delete) for the 119 tracked docs files. Central DB uses WAL mode for concurrent read/write safety.

## Scope
### Must have
- Shared embedding utility at `extensions/autodev/embeddings.ts` — extracts `voyageEmbed()`, `onnxEmbed()`, `embed()`, `EmbedFn` type, `VOYAGE_BATCH_SIZE` constant from `docs/index.ts` into a standalone module. Docs module imports from it instead of defining inline. Loreguard and any future embedding-using system import from it. Magic Context's own embedding system is untouched (external package).
- Shared FTS5 utility at `extensions/autodev/fts-utils.ts` — extracts `checkSqliteVersion()`, `compareVersions()`, `SQLITE_MIN_VERSION` from `loreguard/schema.ts` into a standalone module. Adds `ftsMatchQuery(db, tableName, query, limit)` helper that builds and executes `SELECT ... FROM <table> WHERE <table> MATCH ? ORDER BY rank LIMIT ?` (the FTS5 MATCH + ORDER BY rank pattern shared by both Loreguard and Docs). Loreguard's trigger-based external-content sync stays in Loreguard (domain-specific). Docs' manual insert sync stays in Docs (domain-specific). Only the version check and MATCH query builder are shared.
- Central docs store at `join(getAgentDir(), "..", "docs-corpus")` with co-located `vectors.db` (schema-only on install, populated by seeding). Central home = `getAgentDir()/..` (respects `PI_CODING_AGENT_DIR` override, same as `config-defaults.ts:76`).
- Project docs store at `<project>/docs-corpus/` with DB at `<project>/.autodev/embeddings/vectors.db` (existing path, unchanged).
- `search_docs` queries both DBs, merges results via hybrid search (dense cosine similarity + BM25 FTS5 keyword search via Reciprocal Rank Fusion, 0.7 dense / 0.3 BM25 weight), ranks by fused score, returns top-N with `doc_path` prefixed by tier (`central:pi/sdk.md` vs `project:ARCHITECTURE.md`).
- `docs_status` reports both tiers separately (central chunk count + project chunk count).
- `docs_rebuild` takes a tier parameter: `central` rebuilds from seeded corpus, `project` rebuilds from `<project>/docs-corpus/`.
- CLI sub-commands: `autodev docs rebuild central`, `autodev docs rebuild project`, `autodev docs query <text>` replacing the current stubs. Invalid tier → error + exit 1.
- Seeding framework: pluggable source list (default empty), three source types: `git-sparse` (git clone + sparse checkout), `llms-txt` (parse llms.txt index → fetch .md pages, resolve relative URLs against base), `llms-full` (single-file full docs, `file://` uses `readFileSync`, `http(s)://` uses `fetch`). Per-source `excludePatterns` for filtering junk (glob via `minimatch`). Per-source `active: boolean` flag (default false, source of truth is the YAML config, not the DB). Partial failure: successful sources still rebuild, failed sources logged. YAML parsing via `js-yaml` (add as dependency).
- Lazy refresh: `searchDocsBoth` checks if any central source is stale before querying. Stale = `last_seeded_at` exceeds refresh interval (active sources = 7 days, inactive = 30 days). `active` read from `SeedSource` config (source of truth), falling back to `seed_metadata.active` only if config unavailable. Stale sources refreshed in background (fire-and-forget with `.catch()` logging, non-blocking) — search returns current results immediately, refreshed docs appear on next search. Per-source metadata stored in `seed_metadata` table in central DB ONLY (separate `createCentralDbSchema` function, NOT in `openVectorStore`): `source_name TEXT PRIMARY KEY, last_seeded_at TEXT NOT NULL, commit_hash TEXT, etag TEXT, active INTEGER NOT NULL DEFAULT 0`. NULL hash/ETag = "never seeded" (treat as stale); empty string = "source emits no hash, skip comparison". Missing row for a source present in config = treat as stale (must seed). ETag-and-Last-Modified-absent fallback: fetch body, compute SHA-256, compare to stored `etag` field. Hash/ETag comparison detects actual updates — no re-seed if unchanged. Central DB uses WAL mode (`PRAGMA journal_mode=WAL; PRAGMA busy_timeout=5000`) for concurrent read/write safety. Per-source rebuild: `chunks` table gets `source_name TEXT` column; `rebuildSource(db, sourceName, corpusRoot, embedFn)` clears only chunks matching that source, re-embeds just those files (no full-DB wipe on single-source refresh).
- Agents can add sources to `~/.AutoDev/config/docs-sources.yaml` when they discover new dependencies not covered by the central store.
- `install-module.ts` creates central docs dir + empty `vectors.db` (schema only) during config phase via `openVectorStore()` from `docs/index.ts`.
- `init-module.ts` creates `<project>/docs-corpus/` (empty dir) during `autodev init`.
- `docs-corpus/` in AutoDev repo added to `.gitignore`, 119 tracked files removed from git. `npm pack --dry-run` verifies no docs ship.
- Central seeding source config at `join(getAgentDir(), "..", "config", "docs-sources.yaml")` (central, NOT project-local).
- All existing docs tests updated for dual-tier, new tests for merge search, tier-aware rebuild, and seeding.
