## TL;DR (For humans)

**What you'll get:** Core foundation: clean workspace, preserve `src-agents/` identities, set up project dependencies, port 13 crew agents to pi format (12 from existing `src-agents/*.yaml` + 1 newly-authored Explore), build the base AutoDev extension with all 15 modules (4 foundation modules with real logic: team-mode, comment-checker, intent-gate, notepad; 11 stub modules for later sub-plans), and write Magic Context integration tests with mocks.

**Why this approach:** Pi provides in-process sessions, custom tools, lifecycle events, and extensions with a minimalist philosophy that aligns with AutoDev's autonomy goal. Magic Context already has a Pi extension, so semantic memory comes for free.

**Effort:** L — 6 todos across 2 waves.
**Risk:** High — this is the foundation everything else builds on. Mitigated by: existing design specs, Magic Context handling memory, and pi's proven extension API.

- **Context injection.** Load AGENTS.md, CONTEXT.md, .autodev/memory/*.md via pi's DefaultResourceLoader with agentsFilesOverride. Inject .autodev/reference/ docs as virtual context. Verify agents see the Nautilus crew operating protocol in their system prompt.
- **Team mode (adapted for AutoDev).** Parallel multi-agent coordination with shared mailbox, tasklist, and member management. Adapted for AutoDev's workflow: (a) hyperplan (5 hostile critics) initiated when Harbor Master onboarding completes — the crew critiques the onboarding results before any work begins; (b) background agent manager is "always watching" via team mode — team members observe ongoing work and can flag issues; (c) other agents can chime in with questions/comments during Harbor Master onboarding via team mode (without interrupting the conversation flow) — e.g., Conseil can suggest knowledge-base queries, Metis can flag ambiguities, Momus can challenge assumptions, all through the team mailbox rather than interrupting. 12 team_* tools: team_create, team_delete, team_shutdown_request, team_approve_shutdown, team_reject_shutdown, team_send_message, team_task_create, team_task_list, team_task_update, team_task_get, team_status, team_list.
- **Comment checker.** Strips AI-slop from comments after edit/write tool calls. Implemented as a tool_call event handler (alongside guardrails in T7) that inspects write/edit results and flags AI-generated comment patterns (e.g., "This function does X" → replace with meaningful comment or remove).
- **Notepad system (Loreguard-backed).** Wisdom accumulation across subagent tasks — learnings, decisions, issues, verification, problems. Use Loreguard as the backend: store learnings as ARCHITECTURE memories (via ctx_memory), decisions as ADRs (via Loreguard), issues as CONSTRAINTS memories, verification as evidence files, problems as research notes. This integrates the notepad into AutoDev's existing knowledge systems.
- **IntentGate.** Analyzes true user intent before classifying or acting. Applied in two places: (a) Harbor Master onboarding — IntentGate analyzes the user's initial project description to surface hidden intentions and suggest probing questions; (b) Nemo triage — IntentGate analyzes GitHub issue text to detect the true intent (bug vs feature vs refactor vs question) before Cynefin classification. Prevents literal misinterpretation. Implemented as a pre-processing step in the dispatch pipeline.
- **Built-in MCPs (adapted).** Integrate Context7 (official library documentation lookup) and Grep.app (GitHub code search) as pi tools or MCP integrations. Do NOT integrate Exa web search — agents use pi's built-in web search or ollama web search instead. Context7 helps the crew look up dependency docs during planning. Grep.app helps the Explore agent find real-world code examples.
- **LSP integration.** Register LSP tools via defineTool(): lsp_diagnostics (get errors/warnings), lsp_goto_definition, lsp_find_references, lsp_prepare_rename, lsp_rename, lsp_symbols (file outline/workspace search). These give agents IDE precision. Requires an LSP server running (configured via .pi/lsp.json or auto-detected).
- **Tmux integration.** Interactive bash via tmux panes, team mode visualization (each team member gets a tmux pane), parallel agent monitoring. Register an `interactive_bash` pi tool that creates/manages tmux sessions. Team mode uses tmux for visual layout when tmux_visualization is enabled.
- **Magic Context feature configuration.** Enable and configure these Magic Context features in .pi/magic-context.jsonc: (a) git commit indexing — `memory.git_commit_indexing.enabled: true` for semantic search over git history; (b) key files pinning — `dreamer.pin_key_files.enabled: true` to pin frequently-read files into context (requires file-read tracking); (c) sidekick agent — `sidekick.enabled: true` for prompt augmentation via memory retrieval; (d) workspace memory sharing — configure if the project belongs to a workspace; (e) user memories — enabled by default in Magic Context, extracts behavioral patterns about the user. (f) Dependency docs import — when AutoDev's Harbor Master downloads dependency documentation during onboarding, import it into Magic Context as ARCHITECTURE/CONFIG_VALUES memories via ctx_memory so the crew can search it semantically.
- **Rules injection (.omo/rules/).** Load project-specific rules from .omo/rules/ directory into agent context alongside AGENTS.md. Rules are Markdown files with project-specific coding standards, conventions, and constraints. Injected via the context injection system (T5).
- **Harbor Master as sole user-facing contact.** Harbor Master is the ONLY agent the user interacts with. All other agents are invisible. If any agent needs clarification or hits a blocker, it alerts Harbor Master via the team mailbox. Harbor Master contacts the user via CLI or Discord. Harbor Master is a permanent interface, not just an onboarding agent.
- **Watch Officer proactive monitoring.** The Watch Officer monitors implementation in real time to detect deviations before they happen: plan deviations, API mismatches, dependency incompatibilities, incorrect agent assumptions. This is a proactive role, not just reactive self-healing. The Watch Officer flags issues through the team mailbox before they propagate into committed code.

### Must NOT have (guardrails, anti-slop, scope boundaries)

- **Must NOT migrate any existing code.** Fresh start only. No porting of src/plugin/, src/core/, src/cli/, src/discord/, etc. These are deleted.
- **Must NOT install or depend on OmO (oh-my-openagent).** Build directly on pi primitives.
- **Must NOT depend on OpenCode packages.** Zero OpenCode dependencies.
- **Must NOT reimplement semantic memory.** Magic Context Pi extension provides ctx_search, ctx_memory, ctx_note, ctx_expand, ctx_reduce, historian, dreamer. Install it, don't rebuild it.
- **Must NOT build a single binary.** Future wave. Out of scope.
- **Must NOT make the liaison role mandatory.** The liaison is optional — it applies only when the project is consumed by other agents (e.g., an MCP for Openclaw agents). For human-consumed projects, the crew coordinates deployment directly.
- **Must NOT delete identity/design artifacts.** src-agents/, .autodev/reference/, .autodev/config/, .autodev/memory/, .autodev/skills/, docs-corpus/, AGENTS.md, CONTEXT.md are the design to implement. They survive the fresh start.
- **Must NOT delete or alter preexisting config files.** The user's global `~/.config/opencode/`, `~/.local/share/cortexkit/magic-context/`, and any other global developer tool configs are NOT touched. AutoDev's pi-based setup lives in `.pi/` and coexists with any existing `.opencode/` setup in the same project. No preexisting connections between Magic Context, OpenCode, or other tools are broken.
- **Must NOT delete the existing Magic Context DB.** The 88MB SQLite DB at `~/.local/share/cortexkit/magic-context/context.db` contains real session data and is shared across harnesses. The Pi extension reuses it.
- **Must NOT push to `main`.** All work is on the `pi-foundation` branch. `main` is frozen. No commits, no pushes to `main` during this work. Upon completion, `main` will be deprecated and `pi-foundation` becomes the new `main`.
