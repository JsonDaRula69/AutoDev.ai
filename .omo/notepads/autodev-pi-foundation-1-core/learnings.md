## Task 5: Build base AutoDev pi extension (skeleton + context injection + 4 foundation modules)

- **Extension entry point shape:** `export default function(pi: ExtensionAPI): void` — synchronous factory. Pi waits for async factories before startup, but sync is fine when all module `register()` calls are sync. The factory registers a `before_agent_start` handler for context injection, a `/autodev` command via `pi.registerCommand("autodev", {description, handler})`, and calls `register(pi)` on all 15 modules in canonical order.
- **Context injection strategy:** The extension CANNOT construct `DefaultResourceLoader` (sessions are created by pi, not the extension). The correct approach is `pi.on("before_agent_start", ...)` — the event provides `event.systemPrompt` (chained across handlers) and `ctx.cwd`. Return `{ systemPrompt: augmented }` to inject context. This augments pi's default AGENTS.md loading rather than replacing it. `loadContextFiles(projectRoot)` reads AGENTS.md, CONTEXT.md, and `.autodev/memory/*.md`; missing files are silently skipped so fresh checkouts load cleanly.
- **TypeBox Union + Optional precedence trap:** `Type.Optional(Type.Union([...]), {description})` is a TYPE ERROR — tsc parses the `{description}` as a 2nd arg to `Type.Optional` (which takes only 1 arg), not to `Type.Union`. Fix: `Type.Optional(Type.Union([...], {description}))` — move the options object INSIDE the `Type.Union` call parens. This bit 5 schema definitions in team-mode/schemas.ts. Runtime worked fine (bun is permissive); only `tsc --noEmit` caught it.
- **TypeBox is v1.1.38 (not sinclair/typebox):** The `typebox` package (no `@sinclair` scope) is at v1.1.38. `Type.Union(anyOf, options?)` — options is `{description?, ...}` (TSchemaOptions). `Type.Optional(type)` takes exactly 1 arg. `Static<typeof schema>` extracts the TS type. Imports: `import { Type, type Static } from "typebox"`.
- **pi notify severity strings:** `ctx.ui.notify(message, type?)` accepts `"info" | "warning" | "error"` — NOT `"warn"`. comment-checker initially used `"warn"` and tsc caught it.
- **Test import extensions with verbatimModuleSyntax:** Static imports ending in `.ts` fail TS5097 under `verbatimModuleSyntax: true` unless `allowImportingTsExtensions` is set. Use `.js` extensions in static imports (bun resolves them to the .ts source at runtime). Dynamic `import(join(..., "index.ts"))` with variable strings is fine — tsc doesn't flag dynamic imports with non-literal paths.
- **bun test auto-discovery:** `bun test` auto-discovers files with `.test.`, `_test_`, `.spec`, or `_spec_` in the filename. T4's `test/agent-load.ts` (no infix) is NOT auto-discovered — run with `bun test ./test/agent-load.ts`. T5's test files all use `.test.ts` infix and are auto-discovered.
- **team-mode split:** 12 tools × (schema + executor + registration) would blow the 250 LOC ceiling in one file. Split into 3 files: `schemas.ts` (138 LOC, TypeBox schemas), `executors.ts` (141 LOC, stub execute fns), `index.ts` (143 LOC, register() calls). Each file owns one concept and stays well under the ceiling.
- **comment-checker does NOT block writes:** The `tool_result` handler runs AFTER the write completes. It inspects the content that was written, runs `stripSlop()`, and emits `ctx.ui.notify` warnings. Returning `undefined` from the handler leaves the result unchanged. The write itself is never blocked — slop is reported, not prevented. This matches the spec: "Does NOT block the write — strips slop and lets good code through."
- **intent-gate is pure (no LLM):** Both `analyzeOnboardingIntent()` and `analyzeIssueIntent()` are keyword+structure based, deterministic, and fast. No `pi` API calls, no model invocations. This makes them unit-testable without mocking. A later sub-plan can swap in a model-backed analyzer behind the same return type (`IssueIntentAnalysis` / `OnboardingIntentAnalysis`).
- **notepad file-writing backends:** `storeVerification` and `storeProblem` write files immediately (to `.omo/evidence/` and `.autodev/research/` respectively) and return `written: true` with the absolute path. `storeLearning`, `storeDecision`, `storeIssue` return `written: false` because their backends (ctx_memory, loreguard) are agent tools, not extension APIs — the caller is responsible for invoking them. The descriptor's `note` field tells the caller exactly what to do.
- **Pre-existing typecheck errors:** 8 tsc errors in test/agent-load.ts (7) and test/import-check.ts (1) pre-exist from T3/T4. Confirmed via `git stash` comparison — my T5 files add 0 new errors. These are `noUncheckedIndexedAccess` strictness issues and a missing .d.ts for @cortexkit/pi-magic-context. Not in scope for T5 to fix.

## Task 6: Magic Context integration tests with mocks

- **Mock fixture design:** `test/mocks/magic-context.ts` exports 5 `mock()` functions from `bun:test` — one per ctx_* tool (search, memory, note, expand, reduce). Each returns a canned shape echoing inputs so tests can assert on call args. A `resetCtxMocks()` helper clears call history for `beforeEach` isolation, and `ALL_CTX_MOCKS` tuple enables aggregate assertions. Typed result interfaces (CtxSearchResult, CtxMemoryResult, etc.) keep the fixture type-safe under `strict + noUncheckedIndexedAccess`.
- **No real DB in tests:** Mocks only. The fixture never touches SQLite, the historian, or the dreamer — exactly per the MUST NOT constraints. Tests verify the AutoDev extension's *interaction contract* with the ctx_* tool interface (call args, return shapes), not Magic Context's internals.
- **JSONC parsing trap:** `.pi/magic-context.jsonc` contains the embedding endpoint URL `https://api.voyageai.com/v1`. A naive `text.replace(/\/\/.*$/gm, "")` comment stripper treats `://` as a line comment and truncates the URL mid-string, producing "Unterminated string" JSON parse errors. The fix is a string-aware state-machine stripper that tracks `inString` and only strips `//` / `/* */` outside string literals. This bit 7 config tests on the first run.
- **Config override verification (8 fields, all PASS):** dreamer.model=glm-5.2:cloud, dreamer.fallback_models includes deepseek-v4-pro, embedding.api_key=${VOYAGE_API_KEY}, memory.git_commit_indexing.enabled=true, dreamer.pin_key_files.enabled=true, sidekick.enabled=true, dreamer.user_memories.enabled=true. All read from `.pi/magic-context.jsonc` at test time — no hardcoded expectations.
- **Notepad-to-MagicContext routing verified:** All 5 notepad store* functions produce StorageDescriptors with 5 DISTINCT backends (Set.size === 5): ctx_memory:ARCHITECTURE (learning), loreguard:adr (decision), ctx_memory:CONSTRAINTS (issue), evidence-file (verification), research-note (problem). ## Task 8: Background agent manager + model fallback chains

- **BackgroundManager architecture (8 files, all <250 LOC):** Split into types.ts (115), classifier.ts (88), fallback.ts (191), concurrency.ts (48), circuit-breaker.ts (69), error-handler.ts (60), manager.ts (249), index.ts (51). The manager orchestrates; the other modules own one concept each. Initial manager.ts was 374 LOC — extracted concurrency loader, circuit breaker, and error handler to get under the 250 ceiling.
- **Circuit breaker is a sliding-window timer, not a counter:** Arm on every event, trip when no event arrives for staleTimeoutMs. Idempotent trip: if agent_end arrives concurrently with the trip callback, the event wins (task marked completed) — the trip's onTrip callback sees terminal status and no-ops. Implemented as a separate CircuitBreaker class with an injectable TimerFactory so tests use a manual scheduler (no real setTimeout).
- **Concurrency queue matches by taskId, not by config:** The queue carries {taskId, providerKey}. drainQueue() starts the exact queued task when its provider slot frees — no fragile model-string matching. The task is already in this.tasks as pending; the queue just remembers which pending task to start.
- **Error classifier: unknown → non-retryable (safe default):** classifyError returns {retryable: false, reason: "unknown-non-retryable"} for unrecognized errors. This prevents blind retry loops on errors the system doesn't understand. Retryable: 429, 500, 502, 503, 504, timeout messages. Fatal: 401, 403, context-overflow messages.
- **Fallback is two-stage: proactive chain first, reactive allowlist second:** resolveFallbackModel checks the agent's fallback_models chain (from .pi/agents frontmatter or .autodev/config/fallback.json). If the chain is exhausted or absent, it picks the next untried model from the allowlist (.autodev/config/models.json). Non-retryable errors never trigger fallback — the error surfaces.
- **exactOptionalPropertyTypes caught a real bug:** FallbackOptions.config was `readonly config?: ResolvedFallbackConfig` but the manager passes `config: this.fallbackConfig` where fallbackConfig is `ResolvedFallbackConfig | undefined`. Under exactOptionalPropertyTypes, `?` means "omittable" but NOT "explicitly undefined". Fix: `readonly config: ResolvedFallbackConfig | undefined` (no `?`).
- **Mock fixture pattern:** test/mocks/pi-session.ts exports MockSession (controllable event emitter with abort/dispose tracking) + mockCreateAgentSession (bun:test mock() returning a SessionFactory) + mockSessionRegistry (array of all sessions created in the current test). Tests look up sessions by index, emit events, and assert on manager state. resetSessionMocks() clears the registry in beforeEach.
- **Test timing: fake timers via injectable TimerFactory:** The manager accepts a `setTimer` option. Tests inject a FakeTimerScheduler that records timers without firing them. scheduler.advance(ms) fires due timers in insertion order. No real setTimeout in tests — circuit breaker test is instant.
