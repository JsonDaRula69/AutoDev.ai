
## Task 9: Delegation — category system + task tool for crew dispatch

- **Module split by responsibility (5 files, all <250 pure LOC):** delegation/ split into schemas.ts (20 LOC, TypeBox schema), categories.ts (120 LOC, built-in + custom category map), agents.ts (91 LOC, .pi/agents frontmatter loader), executor.ts (197 LOC, routing + validation + spawn), index.ts (40 LOC, pi.registerTool wiring). Total 428 LOC across 5 files — each owns one concept. executor.ts is the largest at 197 LOC; the foreground-wait polling + all error paths live there.
- **TaskSpawner interface decouples executor from BackgroundManager:** executor depends on a minimal `TaskSpawner` interface (`spawn(config): string`, `getTask(id): {status, result, error} | undefined`), NOT on the full `BackgroundManager` class. Tests inject a `FakeSpawner` that records spawns and lets the test mark tasks complete. Production wires `getBackgroundManager()` (the T8 singleton). This makes the executor unit-testable without standing up real sessions or the circuit breaker.
- **Mutual exclusivity cannot be expressed in TypeBox schema:** `Type.Optional(category)` + `Type.Optional(subagent_type)` allows both-or-neither at the schema level. The executor enforces "exactly one" at runtime: both set → `mutual-exclusivity-violation`, neither set → `missing-dispatch-target`. Schema validation handles type shape; the executor handles semantic constraints.
- **Anti-re-delegation is structural, not flag-based:** `buildTools(baseTools)` filters out `"task"` from every spawned session's tool list. The spawned session literally does not have the `task` tool available — it cannot call something that isn't registered. This is stronger than a runtime check because the LLM never sees the tool in its available tools list.
- **load_skills accepted-but-ignored:** The schema validates `load_skills` as `Type.Optional(Type.Array(Type.String()))`, so the LLM can pass it. The executor references it with `void load_skills` (to satisfy noUnusedLocals) but never injects it into the system prompt. The task spec explicitly says "silently ignored (no skill context injected)" — this matches.
- **loadModelAllowlist reused from T8:** The executor imports `loadModelAllowlist` from `../background/fallback.js` rather than duplicating the allowlist loader. T8 already has a tested, config-reading allowlist loader. No need to reinvent.
- **Foreground wait is bounded (30s max):** When `run_in_background` is false (default), `waitForCompletion()` polls `spawner.getTask()` every 10ms up to FOREGROUND_MAX_WAIT_MS (30s). If the task hasn't completed by then, the tool returns "still running" with the task ID so the caller can poll later. This prevents a stuck child session from hanging the parent's tool call forever.
- **Test microtask-flush pattern for foreground waits:** Tests that exercise the foreground path (run_in_background=false) must let the spawner create the task first (via `await new Promise(r => setTimeout(r, 0))` to flush microtasks), THEN call `spawner.complete(id)` to mark it done, THEN `await execPromise` to get the result. Without the microtask flush, the task isn't created yet when the test tries to complete it — race condition.
- **TypeBox v1.1.38 Optional + Array:** `Type.Optional(Type.Array(Type.String(), { description }))` — the options object goes INSIDE the `Type.Array` call, same pattern as the Union+Optional trap from T5. `Type.Optional` takes exactly 1 arg.
- **Category system is config-driven, not hardcoded:** The 8 built-in categories live in `categories.ts` as `BUILTIN_CATEGORIES`. `loadCategoryMap(projectRoot)` reads `.autodev/config/categories.json` if it exists and merges it over the built-ins (custom entries override built-ins with the same name; new entries are added). The executor never hardcodes a model string — it calls `getCategory()` which reads from this merged map.
